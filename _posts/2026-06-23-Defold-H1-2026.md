# Defold - summary of the first half of 2026

The first half of 2026 has been an important period for Defold. Across five releases in the 1.12 series - [1.12.0](/2026/01/12/Defold-1-12-0/), [1.12.1](/2026/02/04/Defold-1-12-1/), [1.12.2](/2026/03/02/Defold-1-12-2/), [1.12.3](https://forum.defold.com/t/defold-1-12-3-has-been-released/82619), and [1.12.4](/2026/05/04/Defold-1-12-4/) - we continued improving the engine, editor, web runtime, rendering pipeline, 3D workflows, extensions, examples, and documentation. The period ends with [Defold 1.13.0](https://forum.defold.com/t/defold-1-13-0-has-been-released/82980/12), one of the larger modernization releases. It brings, among other things, new Box2D scripting APIs, [Morph Targets support for 3D models](/manuals/model-animation/#morph-targets), rendering improvements, and a broad removal of deprecated APIs.

Earlier this year, in [A look ahead for 2026](/2026/02/10/Defold-A-look-ahead-for-2026/), we outlined several priorities: better editor workflows, improved editor performance, more 3D work, richer animation support, rendering improvements, extension updates, and more tutorials and examples. Looking back at the first half of the year, many of those plans have already turned into concrete releases, and we are very glad to see that progress.

## A better foundation for game logic control

[Defold 1.12.0](/2026/01/12/Defold-1-12-0/) started the year with a change that affects how developers structure gameplay code: the update lifecycle was clarified and extended with the new [`late_update()`](/manuals/application-lifecycle/) function.

The new order is like this:

* `fixed update()` run first as much as needed to ensure fixed timestep;
* `update()` runs once per frame;
* `late_update()` runs after regular update logic and before rendering.

![application_lifecycle](/images/posts/h1-2026/application-lifecycle.webp)

Messages are also flushed between each update phase, which gives developers more predictable control.

This is especially useful in projects where gameplay and presentation should not always happen in the same step. A common pattern is to update game logic first, then update visuals, camera, UI indicators, or other dependent systems. `late_update()` gives Defold projects a cleaner place for that kind of work. Check out more details in the updated [Application Lifecycle manual](/manuals/application-lifecycle/).

## Pause when idle - engine throttling

Defold 1.12.0 also introduced a new [engine throttling API](/ref/sys-lua/#sys.set_engine_throttle:enable-cooldown). It allows a game to temporarily skip full engine updates and rendering while still listening for input. As soon as input is detected, the engine wakes up again for a configurable cooldown period, after which it can return to the throttled state automatically. So instead of rendering and updating continuously when nothing is happening, Defold can now reduce work until the player interacts again.

This is especially useful for games and applications that spend time in mostly static states: menus, idle screens, turn-based games, puzzle games waiting for player input, and great for battery-sensitive mobile and other portable devices.

## A major step for Box2D physics scripting

One of the most exciting parts of Defold 1.13.0 is the expanded [Box2D scripting API](/ref/stable/b2d-lua/).

Defold had always included strong [physics support](/manuals/physics/), but the new Box2D scripting functions give developers much more direct control from Lua. The release adds joint creation, inspection, destruction, body/shape access, fixture and shape manipulation, mass data access, sensor changes, filter updates, AABB queries, joint limits, motors, spring settings, damping, reaction forces, torque, and more. This is powerful because it opens up a wider class of physics-driven gameplay without requiring native code.

With the new APIs, developers can more easily build things like:

* ropes, chains, bridges, cranes, vehicles, and mechanical contraptions;
* grappling hooks and swinging mechanics;

## Morph Targets aka Blend Shapes for 3D animation

Another major 1.13.0 feature is support for [Morph Targets, also known as Blend Shapes](/manuals/model-animation/#morph-targets), in [3D model components](/manuals/model/).

This makes it possible to animate model shapes directly, which might be used for facial expressions, mouth movement, muscle deformation, soft-body-like effects, shape transitions, damage states, character customization, and many other effects that are awkward or impossible to express with skeletal animation alone. With morph target support, Defold's [glTF pipeline](/manuals/importing-models/) becomes more useful for expressive 3D characters and animation-heavy 3D projects.

It is part of a broader 3D cleanup. Defold 1.13.0 also removed Collada support and committed to glTF as the modern 3D asset format. This lets us focus our effort on the pipeline that matters most today instead of maintaining older and unsupported solutions. The release also fixed model transform inconsistencies between editor and runtime for imported glTF assets, making 3D behavior more predictable.

There is still more work ahead, but morph targets are a meaningful milestone for Defold's 3D future. You can try this today in the [Morph Target Animation](/examples/model/morph-target/) and [Morph Target Weights](/examples/model/morph-target-weights/) examples and read the [Morph Targets manual](/manuals/model-animation/#morph-targets).

## Better Camera component

[Camera](/manuals/camera/) came a long way to become a more useful component and the [`camera` API](/ref/camera/) grew. Editor-wise the camera shows the used projection frustum and the camera representation in the editor is scaled properly.

Defold 1.12.0 added camera conversion helpers such as [`camera.world_to_screen()`](/ref/camera/#camera.world_to_screen) and [`camera.screen_to_world()`](/ref/camera/#camera.screen_to_world), making it easier to connect world-space gameplay with screen-space UI or mouse/touch input. See also the [World to Screen](/examples/render/world_to_screen/) and [Screen to World](/examples/render/screen_to_world/) examples.

We also have adaptive zoom - orthographic camera can now be adjusted even when using Fit or Cover mode, and a new [`camera.get_orthographic_auto_zoom()`](/ref/camera/#camera.get_orthographic_auto_zoom:camera) and a read-only `orthographic_auto_zoom` properties were added. And because the camera is selected now with `render.set_camera()` in render script, there's no need to acquire and release camera focus nd those functions which had already been deprecated have now been removed.

Since 1.13.0 you can also preview what a selected camera sees directly in the Editor. This makes it easier to set up the scene or set correct camera position and orientation.

## Going international - font shaping and collections

Beside the [Editor](/manuals/editor/#editor-language) and the documentation being translated to other languages like Chinese, Portuguese and Russian, the engine now also has support for [runtime text shaping and font collections](/manuals/font/) since 1.12.0. This improves how Defold handles more advanced text layout cases, including proper kerning pairs, ligatures, and right-to-left text. It is an important step for localization, especially for games targeting languages and scripts where basic left-to-right character placement is not enough.

The new workflow allows developers to associate multiple `.ttf` files with a single font resource and load only the fonts needed for a given language. It also supports prewarming text so required glyphs can be generated before they are shown on screen, which makes runtime-generated fonts more practical for multilingual games.

## Web games

Web remains one of Defold's strongest areas. In [Defold 1.12.4](/2026/05/04/Defold-1-12-4/), an empty HTML5 project was reduced to around **1.06 MB**. We are following the [size changes](https://github.com/defold/build-size) and are very cautious about it to keep Defold as slim as possible. For browser games, this is not just a nice technical detail. It directly affects loading time, and thus conversion to play, player retention, and also the ability to reach users on a wide range of devices.

We also continued cleaning up the web runtime. Unsupported and unused web functionality were removed, and with 1.13.0, Defold fully commits to `wasm-web` and removes the old `asm.js` target. WebAssembly is the modern and widely supported target for web games, and this lets Defold focus on one stronger, cleaner web path.

## Live Update

[Live Update](/manuals/live-update/) also moved forward. In 1.12.4, legacy Live Update APIs were deprecated, and in 1.13.0, the old auto-mount feature and the old Live Update archive format were removed. The recommended path is now clearer: use [`liveupdate.add_mount()`](/ref/liveupdate/#liveupdate.add_mount) and mount zip archives or folders explicitly.

The new direction gives developers more control over what is mounted, when it is mounted, and how content is prioritized. Developers can keep the first download small, get players into the game quickly, and then download additional content.

Live Update archives can also be mounted from Android assets, which is useful for asset delivery strategies, and can be added to the Android bundle either through bundle resources or as an install-time [Play Asset Delivery](
https://forum.defold.com/t/android-play-asset-delivery/82174) pack.

## Sounds good

While the sound streaming already existed in 2025, this year we've pushed sounds forward as well, and introduced a new AVAudio backend for iOS/macOS, replacing the deprecated OpenAL backend.

In 1.12.1 we've added Windows sound device recovery and improved how raw sounds are handled in the editor by wrapping them as embedded sound components.

In 1.12.4 we've improved the internal sound buffers so they can grow dynamically, fixing crashes with high sample rates or high playback speed, increasing the maximum speed limit from 5 to 50, and adding a new Sound System section to the profiler.

Finally, we've also released the [official FMOD extension for Defold](https://forum.defold.com/t/fmod-extension-for-defold-now-official/82973), which gives users with more advanced sound system requirement the capabilities they need for their sound design.

## Rendering and graphics keep moving forward

In 1.12.2, model components gained [custom vertex attribute data](/manuals/material/#setting-custom-vertex-attribute-data) support, allowing developers to set and animate model attributes from Lua. 1.12.3 exposed time to shaders and added a [Texture Transform semantic](/manuals/material/#attributes), which made examples such as [texture scrolling](/examples/sprite/texture_scrolling/) or [true billboarding](/examples/material/billboarding/) easier and cleaner.

Defold 1.12.4 added shader precision options and increased texture units per draw from 8 to 16. This gives developers more room for complex materials and multi-texture effects, especially in 3D or advanced 2D rendering pipelines.

With 1.13.0, [Vulkan](https://www.vulkan.org/) is the default graphics API on Android, with OpenGL available as fallback. This release also improved and expanded blending options and added more [shader constants](/manuals/material/#vertex-and-fragment-constants), including commonly used inverse matrix variants. New [`graphics`](/ref/graphics/) script functions expose information about the active graphics adapter and supported features. This allows game code and render scripts to make better decisions based on the current device and backend. It is especially useful for cross-platform games where rendering features may need to adapt between desktop, mobile, and web. New [`material`](/ref/alpha/material/) and [`compute`](/ref/alpha/compute/) script modules make it possible to inspect and update material and compute resources from Lua.

[Particle FX components](/manuals/particlefx/) now support [frustum culling](/manuals/render/#frustum-culling), which might give a boost for games with a lot of particle effects.

Defold's rendering system has always been flexible through [render scripts](/manuals/render/) and [materials](/manuals/material/), but these changes expose more of that power to game code.

## Cleaner APIs

Defold 1.13.0 also removed a large number of deprecated Lua APIs and old constants.

This kind of cleanup can require migration work in older projects, but it is important for the long-term health of the engine. A small engine should also have a clean API. Removing deprecated functions, clarifying module boundaries, and consolidating graphics constants makes Defold easier to maintain, easier to document, and easier to teach.

It also gives us a better foundation for future features.

## Automated and custom workflows

We also improved the `/command/build` endpoint, so external tools can know when a build has actually completed. In 1.12.3 we've added OpenAPI support for the Editor HTTP Server, which makes the editor server easier to discover and use from external programs. And in Defold 1.13.0 we introduced a new `/eval` endpoint for the editor server, which allows developers to design more automation systems - these features help users who want to incorporate agents into their workflow to close the loop, build more advanced toolchains, including local automation e.g. for testing, and agentic systems that can inspect a project, run editor commands, evaluate small operations, trigger builds, and help developers with repetitive production tasks. We won't be forcing AI-assisted workflows, but we'll try to make development convenient and reliable for all users - those who want to use use LLMs, and those wanting to create custom automations.

We want the editor to stay lightweight, but also become more customizable and easier to integrate into each team’s workflow. Instead of forcing every project into the same large editor pipeline, Defold should provide reliable extension points, and allow further customizations and automations for game creation, and we will keep moving in this direction.

## Extensions continued to evolve

Extensions remain central to Defold's design - we keep the engine lightweight, while powerful external integrations can be easily added to projects.

The [Rive extension](/extension-rive/) got major updates, including a new data binding API, a Lua binding overhaul, generated documentation, async APIs, global listeners, Linux Vulkan backend support, and later Luau scripting support.

The [Spine extension](/extension-spine/) received fixes and improvements for empty first frames, property options, indexed-buffer rendering, track alpha support, Linux ARM64 support, and deprecated API cleanup.

Other official extensions also moved forward, including [AdMob](/extension-admob/), [WebView](/extension-webview/), [Firebase Remote Config](/extension-firebase-remoteconfig/), [IronSource](/extension-ironsource/), [Push](/extension-push/), [WebSocket](/extension-websocket/), and the mentioned FMOD and Firebase Crashlytics extensions.

This extension work is important because many production games depend on ads, analytics, platform SDKs, animation runtimes, web views, remote configuration, push notifications, and platform-specific integrations. We believe that keeping these pieces moving alongside the engine is part of keeping [Defold production-ready](/product/).

## A strong first half of 2026

The first half of 2026 moved Defold forward in many ways.

The engine became smaller and cleaner. The editor became faster and more capable, with 3D workflows advanced. Morph targets opened the door to more expressive 3D animation. Box2D scripting gave developers much deeper runtime control over physics. Rendering gained new low-level tools. Live Update became more explicit and predictable. Extensions continued to evolve. More examples, guides, games, and community tools appeared. And two new partnerships strengthened the ecosystem around the engine.

There is still more to do, of course. But the direction described in the [2026 roadmap](/2026/02/10/Defold-A-look-ahead-for-2026/) is already visible in the releases.

The second half of 2026 should be just as exciting. Stay tuned!
